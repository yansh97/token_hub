# Token Hub 前端重构计划

状态：原则已确认，等待页面审计与设计阶段启动

日期：2026-07-20

## 1. 背景

Token Hub 前端长期继承上游 Token Proxy 的页面结构与组件实现，并叠加了多轮 UI/UX 调整。目前存在以下问题：

- 历史模板组件、未使用依赖和重复图标库仍留在仓库中。
- 简单页面使用了超出实际需求的路由、表格和国际化基础设施。
- 页面布局、字体层级、间距、控件密度和信息组织缺少统一规则。
- 部分数据展示直接继承后端字段结构，没有围绕用户任务重新组织。
- 测试中存在对旧 DOM 结构和 Tailwind class 的强耦合，不利于重构。
- 前端文件与上游数据契约边界不够清楚，增加后续同步成本。

本次工作不是对旧 UI 换皮，而是在保持后端能力、数据契约和上游同步路径稳定的前提下，重新设计并实现前端。

## 2. 总体目标

### 2.1 产品目标

- 围绕仪表盘、提供商、日志和设置四类核心任务重新设计信息架构。
- 建立适合桌面运维工具的现代化、克制、紧凑的视觉系统。
- 参考 Codex、ChatGPT 和 OpenAI 工具界面的设计语气，但不直接复刻。
- 提升信息扫描、配置编辑、状态判断和日志排障效率。
- 保持加载、空状态、错误、禁用、确认和反馈等完整交互状态。

### 2.2 工程目标

- 后端无需修改即可运行重构后的前端。
- 保持 Tauri command、事件、配置和数据结构兼容。
- 尽可能减少外部依赖，优先使用 React、原生 HTML、浏览器 API 和 CSS。
- 删除历史模板文件、未使用依赖和重复抽象。
- 建立清晰的 UI primitives、应用公共组件和业务组件边界。
- 重构前端测试，使其验证用户行为和数据契约，而不是旧实现细节。
- 将与上游容易冲突的修改限制在必要范围内。

## 3. 非目标

- 不修改 `src-tauri/**` 中的后端实现。
- 不改变代理转发、格式转换、负载均衡、计费、统计和账号逻辑。
- 不修改数据库结构、持久化格式或已有配置语义。
- 不为了新 UI 要求后端增加展示专用字段。
- 不原样复刻当前页面、控件布局或卡片结构。
- 不建立第二套长期并存的 `src-v2` 应用。
- 不引入全局状态库、请求缓存框架或大型表单框架作为默认方案。
- 不为推测中的未来功能预留复杂抽象。

## 4. 已确认的硬约束

### 4.1 后端与数据兼容

- `src-tauri/**` 视为不可修改边界。
- Tauri command 名称、参数、返回结构和调用语义保持不变。
- Tauri 事件名称与 payload 结构保持不变。
- 数据库、配置文件和持久化格式保持兼容。
- `src/features/**/api.ts`、原始数据类型和数据 Hook 可以在必要时调整，但必须保持调用行为一致或向后兼容。
- 原始契约类型不得混入格式化文本、视觉状态或组件专用字段。
- UI 需要的新字段优先通过纯函数从已有数据派生。
- 配置读写必须保持 round-trip 兼容，不能意外丢弃未知字段。
- 对 API、类型和 Hook 的修改必须具有明确收益，不做纯风格重构。

### 4.2 上游同步

- 后端能力、API 契约和数据类型继续以上游同步为优先。
- 上游新增字段时，前端应默认容忍并按需展示。
- 上游调整契约时，优先修改展示适配层，避免把 UI 逻辑写入原始类型。
- UI/UX 改动尽量集中在前端展示边界内。
- 不批量移动、改名或格式化上游高频变动的数据文件。

### 4.3 单语言

- 产品只保留中文。
- 删除 Paraglide、语言 Provider、英文消息和生成消息模块。
- 页面文案直接使用中文；只有跨页面重复且必须一致的领域术语才提取常量。
- 时间、日期和数字统一使用 `Intl`，locale 固定为 `zh-CN`。
- 不保留未被要求的“未来恢复国际化”兼容层。

## 5. 目标技术栈

### 5.1 保留

- React 19
- TypeScript
- Vite
- Tauri 2 及实际使用的 Tauri 插件
- Tailwind CSS 4
- Lucide React
- React Testing Library
- Vitest

### 5.2 新的复杂组件基础层

采用 `@base-ui/react` 作为复杂交互 primitives：

- Dialog / Alert Dialog
- Dropdown Menu
- Tooltip / Popover
- Select / Combobox
- Tabs
- Toast（在覆盖需求和行为验证通过后替代 Sonner）

选择 Base UI 的原因：

- 无样式，适合 Tailwind 和自定义产品视觉。
- 覆盖焦点管理、键盘交互、ARIA、Portal 和弹层定位。
- 单一包替代当前多个 `@radix-ui/react-*` 直接依赖。
- 支持按组件子路径导入和 tree-shaking。
- 可使用 shadcn/ui 的 Base UI 实现作为初始源码参考。

### 5.3 本地简单组件

以下组件使用原生 HTML 和本地源码实现，不依赖复杂 primitives：

- Button / IconButton
- Input / Textarea
- Checkbox / Switch
- Card
- Badge / StatusDot
- Table
- Separator
- Skeleton
- Alert
- EmptyState
- Spinner

shadcn/ui 在本项目中作为组件源码和设计约定参考，不作为运行时组件包。生成的源码必须删除未使用的 variant、`asChild` 和额外依赖。

### 5.4 路由

删除 TanStack Router，使用原生 Hash 路由：

```text
#/dashboard
#/upstreams
#/logs
#/settings
```

实现要求：

- 使用原生 `<a href>`、`hashchange` 和浏览器历史。
- 使用 React `useSyncExternalStore` 订阅当前路由。
- 使用静态 route 到页面组件映射。
- 未知路由回退到仪表盘。
- 保持刷新、前进和后退可用。
- Tauri deep-link 能力与前端页面路由分离处理。

### 5.5 表格

- 默认使用语义化原生 `<table>` 和本地 Table 样式组件。
- 固定表头、滚动、列宽和行状态使用 CSS 实现。
- 分页继续由现有数据能力提供。
- 只有出现复杂排序、筛选、列状态或大数据虚拟化需求时，才重新评估专业表格库。

### 5.6 图表

- 第一阶段暂时保留 Recharts，避免在重构同时重写图表布局、坐标和 Tooltip 行为。
- 页面设计确定后评估图表复杂度。
- 仅在图表足够简单、原生 SVG 实现明显更小且可维护时移除 Recharts。
- 不为了减少一个依赖而手写复杂图表引擎。

## 6. 依赖治理

### 6.1 引入标准

新依赖必须同时满足：

1. 原生机制或少量本地代码无法可靠完成。
2. 存在明确且稳定的实际使用场景。
3. 自行维护成本高于引入和升级成本。
4. 与 React 19、Vite 和 Tauri WebView 兼容。
5. 支持按需导入，不强制引入新的样式运行时。

### 6.2 计划移除

- `@tanstack/react-router`
- `@tanstack/router-cli`
- `@tanstack/router-plugin`
- `@tanstack/react-table`
- `@tanstack/react-virtual`
- `@inlang/paraglide-js`
- `@tabler/icons-react`
- 全部未使用的 `@dnd-kit/*`
- 当前多个 `@radix-ui/react-*` 包（Base UI 迁移完成后）
- `vaul`
- `zod`（当前无实际使用）
- `big.js`（当前无实际使用）
- `sonner`（Base UI Toast 或本地反馈组件验证后）
- `next-themes`（由小型本地主题机制替代后）

删除依赖前必须确认生产代码、测试、构建脚本和 Tauri 入口均无使用。

## 7. 数据与展示架构

目标数据流：

```text
Tauri 后端
  → API / Command 客户端
  → 原始契约类型
  → 数据 Hook
  → selector / view model
  → 页面组件
```

### 7.1 API 层

- 只负责调用 Tauri command、订阅事件和返回原始结果。
- 不包含 Toast、导航、格式化文本或 JSX。
- 保留后端错误信息，不在此层转换为页面文案。

### 7.2 原始类型

- 与后端序列化字段保持对应。
- 可选字段和兼容字段保持明确。
- 不使用组件 props 类型替代后端契约类型。

### 7.3 数据 Hook

- 管理加载、刷新、错误、请求竞态和订阅生命周期。
- 使用事件处理器触发用户操作，不通过 Effect 模拟点击行为。
- Effect 仅用于与 Tauri 事件、计时器或浏览器外部状态同步。
- 不为廉价派生值添加无意义的 `useMemo`。
- 不默认建立全局数据缓存。

### 7.4 View Model

- 使用纯函数完成分组、排序、格式化和展示字段派生。
- 不改变原始对象。
- 不发起网络或 Tauri 调用。
- 便于使用单元测试覆盖边界数据。

## 8. 视觉与交互原则

### 8.1 产品风格

- 中性、克制、内容优先。
- 低装饰、低阴影、低色彩噪声。
- 使用清晰的排版层级和稳定的对齐基线。
- 主要操作明确，次要操作弱化。
- 不使用营销式大标题、渐变背景、装饰性图形和悬浮页面卡片。
- 不使用 Card 包裹整个页面，不创建 Card 嵌套 Card。
- 状态色仅用于成功、警告、错误和运行状态。
- 图标统一使用 Lucide，不手绘已有图标。

### 8.2 初始排版规格

```text
页面标题       18px / 24px
区块标题       15px / 20px
正文和控件     13px / 20px
辅助信息       12px / 16px
紧凑标签       11px / 16px
```

- 不使用随 viewport 连续缩放的字体。
- 中文字体优先使用系统字体：`-apple-system`、`BlinkMacSystemFont`、`Segoe UI`、`PingFang SC`、`Microsoft YaHei`、sans-serif。
- JSON、模型名、路径、ID 和技术字段使用等宽字体。
- letter spacing 默认为 `0`。

### 8.3 初始尺寸规格

```text
间距           4 / 8 / 12 / 16 / 20 / 24 / 32
控件高度       28 / 32 / 36
圆角           4 / 6 / 8
边框           1px 语义色
阴影           仅弹层和必要浮层使用
```

这些值是设计审计的起点，最终以页面原型和跨页面一致性验证为准。

### 8.4 交互规则

- 图标按钮必须有可访问名称；陌生图标提供 Tooltip。
- 二元设置使用 Checkbox 或 Switch。
- 模式切换使用 Segmented Control 或 Tabs。
- 数值使用 Input、Stepper 或 Slider，不用普通文本按钮模拟。
- 破坏性操作需要明确确认。
- 加载、空数据、错误、禁用和只读状态必须进入设计稿。
- 弹层必须支持 Escape、焦点锁定和关闭后焦点恢复。
- 页面和弹层不得出现文字溢出、控件重叠和不可解释的布局跳动。

## 9. 组件分层

```text
src/
  components/
    ui/          无业务含义的基础组件
    app/         跨页面的产品结构组件
  features/
    dashboard/
    logs/
    upstreams/
    settings/
  lib/
    routing/
    theme/
    format/
```

### 9.1 `components/ui`

只包含无业务含义的 primitives，例如 Button、Input、Dialog、Menu、Tooltip 和 Table。

### 9.2 `components/app`

包含 AppShell、Sidebar、PageHeader、Toolbar、EmptyState、ErrorState 等跨页面产品结构。

### 9.3 `features/*`

包含 RequestTable、RequestDetail、ProviderEditor、UsageChart 和 ServiceStatus 等具体业务组件。

### 9.4 抽象约束

- 不创建 `CommonCard`、`UniversalPanel` 或配置驱动的万能组件。
- 只有两个以上页面存在稳定的相同行为时才提取公共组件。
- 不为一次性布局建立 wrapper 组件。
- 每个组件保持小 API、明确命名和单一责任。

## 10. 页面重新设计流程

每个页面在编码前必须完成以下产物：

1. 用户任务清单。
2. 当前 API、Hook、类型和后端字段清单。
3. 主信息、辅助信息和排障信息分级。
4. 高频操作与低频操作分级。
5. 页面信息架构草稿。
6. 桌面 viewport 线框图。
7. 页面组件树。
8. 加载、空状态、错误和边界状态设计。
9. 行为验收清单。

旧页面仅用于确认已有功能和字段，不作为布局模板。

### 10.1 仪表盘

重点：运行状态判断、使用趋势、资源分布和最近请求。避免堆砌等权指标卡。

### 10.2 日志

重点：快速筛选、扫描、状态识别和问题排查。请求详情使用结构化侧边面板或 Dialog，不直接平铺原始数据。

### 10.3 提供商

重点：查看配置状态、添加、编辑、启停和模型管理。列表与编辑流程保持连续，高频字段优先，低频高级字段渐进披露。

### 10.4 设置

重点：服务状态、代理核心配置、存储、启动和更新。使用纵向设置分组与行级控件，避免大量独立卡片。

以上只定义页面任务，不提前锁定最终布局。最终结构必须依据数据审计和设计草稿确认。

## 11. 前端测试重构原则

现有前端测试允许随组件和页面重构一起调整或重写。

### 11.1 保留的测试价值

- 后端调用参数和调用时机。
- 配置序列化与兼容行为。
- 错误、空值和可选字段处理。
- 请求竞态和最新结果优先行为。
- 用户可见的关键数据和操作结果。

### 11.2 应删除或重写的测试

- 对具体 Tailwind class、固定 DOM 层数和旧布局结构的断言。
- 只验证实现细节而不验证用户行为的测试。
- 已删除模板组件的快照和测试。
- 对旧路由框架 mock 的测试。
- 对已移除国际化函数的测试。

### 11.3 测试层级

| 层级 | 目标 |
| --- | --- |
| 契约测试 | command、参数、事件 payload、配置 round-trip |
| 纯函数测试 | selector、view model、格式化和边界数据 |
| Hook 测试 | 加载、错误、刷新、竞态和订阅清理 |
| 组件测试 | 用户操作、可访问名称、状态切换和结果展示 |
| 页面测试 | 页面关键工作流与数据组合 |
| 视觉 QA | 桌面 viewport、浅色/深色、弹层和长内容 |

测试优先使用 role、label、可见文本和用户事件，不依赖 class 名。

## 12. 实施阶段

### 阶段 0：基线与审计

产物：

- 页面、路由、组件和依赖使用清单。
- Tauri command、事件、API 和数据类型契约清单。
- 当前功能与测试覆盖矩阵。
- 可删除文件和依赖清单。
- 四个页面的用户任务和数据字段审计。

验收：

- 未修改后端。
- 当前测试、类型检查和构建结果有明确基线。
- 每个已有功能都有迁移归属。

### 阶段 1：运行骨架

内容：

- 建立原生 Hash 路由。
- 删除 TanStack Router 和生成路由。
- 移除国际化运行时，保留中文。
- 建立本地主题机制。
- 建立新的 AppShell、Sidebar 和页面容器。
- 引入 Base UI，并只实现首批必要 primitives。

验收：

- 四个页面入口、刷新、前进和后退可用。
- Tauri 启动和更新跳转行为保持可用。
- 浅色和深色主题可用。
- 无后端和数据契约变化。

### 阶段 2：设计系统与公共组件

内容：

- 定义颜色、字体、间距、圆角、边框、状态和焦点 token。
- 实现 Button、Input、Table、Badge、Dialog、Menu、Tooltip 等基础组件。
- 实现 PageHeader、Toolbar、EmptyState、ErrorState 等应用组件。
- 建立组件示例页或开发态检查入口，仅用于开发验证，不进入正式导航。

验收：

- 基础组件覆盖全部必要状态。
- 不存在重复 variant 和未使用 props。
- 键盘与焦点行为通过检查。

### 阶段 3：逐页重构

推荐顺序：

1. 仪表盘：建立指标、图表和表格规范。
2. 日志：复用表格规范并建立详情面板规范。
3. 提供商：实现复杂编辑和配置工作流。
4. 设置：统一行级配置和服务状态表达。

每个页面独立完成：数据审计、设计草稿、组件树、实现、测试、视觉 QA 和旧代码清理。

### 阶段 4：依赖与历史代码清理

内容：

- 删除旧页面、旧组件、旧路由和生成文件。
- 删除未使用的 shadcn 模板组件。
- 删除 Tabler、Radix、TanStack、Paraglide、Vaul、dnd-kit 等已替代依赖。
- 检查生产 bundle 中的依赖和重复模块。
- 删除只服务旧实现的测试 helper 和 mock。

验收：

- `package.json` 中每个运行时依赖都有明确生产使用点。
- 无两套图标、路由、国际化或组件体系并存。
- 无旧 UI 兼容层残留。

### 阶段 5：完整回归与发布准备

内容：

- 完整前端测试、lint、类型检查和构建。
- Tauri 桌面运行验证。
- 浅色/深色及主要桌面尺寸视觉 QA。
- 配置读写、更新、启动、日志捕获和提供商编辑回归。
- 与上游契约清单重新核对。

验收：

- 无 P0、P1、P2 视觉或功能问题。
- 后端无需修改即可运行。
- 关键工作流与现有后端完全兼容。

## 13. 迁移纪律

- 按页面垂直切片推进，不同时重写所有页面。
- 新旧页面可以在开发过程中短暂共存，但最终不保留双实现。
- 每个阶段只删除已经被验证替代的代码和依赖。
- 不在页面重构中顺手修改无关后端或数据文件。
- 不把依赖清理、页面重设计和数据契约修改混入无法审查的单次提交。
- 每次提交应能清楚说明行为变化、视觉变化和验证结果。

## 14. 风险与缓解

| 风险 | 缓解 |
| --- | --- |
| 重构遗漏旧功能 | 阶段 0 建立功能与字段迁移矩阵 |
| UI 需求推动后端变化 | 使用 view model，从现有字段派生展示数据 |
| 配置保存丢失未知字段 | 增加配置 round-trip 契约测试 |
| 替换路由影响更新跳转 | 为所有程序化导航建立原生路由测试 |
